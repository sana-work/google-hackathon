APPLICABILITY: All IT systems, infrastructure, applications, and personnel

═══════════════════════════════════════════
SECTION 1 — ACCESS CONTROL
═══════════════════════════════════════════

1.1 Authentication: All production systems require username + password authentication. Multi-Factor Authentication (MFA) is currently implemented for:
    (a) VPN access
    (b) Cloud console access (GCP, AWS)
    — NOTE: MFA is NOT yet enforced for application-level administrative dashboards or database admin tools. This is on the roadmap for Q3 2026.

1.2 Privileged Access: Privileged accounts (root, admin, DBA) use shared credentials stored in a password vault (CyberArk). Access reviews are conducted semi-annually.
    — NOTE: Just-in-time provisioning is not yet implemented. Privileged users have standing access.

1.3 Service Accounts: Service accounts are documented in the CMDB. Credential rotation occurs annually.

═══════════════════════════════════════════
SECTION 2 — ENCRYPTION STANDARDS
═══════════════════════════════════════════

2.1 Data at Rest: AES-256 encryption using Google-managed encryption keys (GMEK) for all production databases and storage.
    — NOTE: Customer-managed encryption keys (CMEK) are used only for the Payment Gateway system.

2.2 Data in Transit: TLS 1.2 minimum for all external communications. Internal service-to-service communication uses mTLS where supported.

═══════════════════════════════════════════
SECTION 3 — INCIDENT RESPONSE
═══════════════════════════════════════════

3.1 Incident Detection: The SOC monitors security events using Splunk SIEM. Alert rules are reviewed quarterly.

3.2 Incident Reporting Timeline:
    (a) Internal escalation: Within 4 hours of detection
    (b) Regulatory reporting: Within 72 hours for incidents involving personal data (per GDPR/DPDPA)
    (c) Board reporting: Quarterly summary for all incidents; immediate briefing for CRITICAL severity

3.3 SOC Operations: The SOC operates during business hours (8 AM — 8 PM IST, Monday-Friday). After-hours monitoring is handled by an automated alerting system with on-call engineer response.
    — NOTE: 24x7 SOC staffing is planned for Q2 2026 but not yet operational.

═══════════════════════════════════════════
SECTION 4 — VULNERABILITY MANAGEMENT
═══════════════════════════════════════════

4.1 Scanning: Automated vulnerability scans run weekly on all internet-facing systems.

4.2 Patching Timelines:
    (a) CRITICAL: Within 7 days of vendor patch release
    (b) HIGH: Within 14 days
    (c) MEDIUM: Within 30 days
    (d) LOW: Within 90 days

4.3 VAPT: External Vulnerability Assessment and Penetration Testing is conducted annually by an independent auditor.

═══════════════════════════════════════════
SECTION 5 — SECURITY LOGGING AND MONITORING
═══════════════════════════════════════════

5.1 All access events, configuration changes, and data export operations must be logged.
5.2 Logs are retained for 180 days in hot storage and 1 year in cold storage.
5.3 Log integrity is protected via write-once storage, but no session recording is currently implemented for privileged access sessions.
